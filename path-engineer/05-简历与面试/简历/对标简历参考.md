## 一、最对口的真实模板：派聪明 RAG 知识库项目

> **这个项目就是你阶段 3-5 准备做的项目的"原型"**——同样的技术栈、同样的功能。
> 你只要按 path-engineer/03-SpringAI与RAG/ 走完，就能写出对标版本。

### 📌 完整项目描述（可直接套用）

```
项目名称 ▏派聪明 - 企业级 RAG 智能知识库系统
角色    ▏Java 后端开发     时间 ▏2024.05 - 2024.10

【技术栈】
Spring Boot 3 + Spring AI + Spring Security + JWT
MySQL + Redis + MinIO + Elasticsearch（IK 分词器）
Kafka + WebSocket / SSE
DeepSeek + 通义千问 Embedding + Ollama

【项目描述】
派聪明是基于私有知识库的企业级智能对话平台，
允许用户上传文档构建专属知识空间，
通过自然语言交互方式查询和获取知识。

【核心职责】
1. 参与整体后端架构设计，基于 Spring Boot + Spring Security
   构建模块化分层架构，确保系统低耦合、高内聚
2. 设计 MySQL + Redis + MinIO + Elasticsearch 的多级存储体系，
   实现文件元数据、向量索引、原文存储分离
3. 利用 Redis BitMap 存储文件分片状态，1000 个分片仅占 125 字节，
   结合 MinIO 实现大文件分片上传与断点续传，
   1GB 文件上传耗时由 15s 降至 3s
4. 利用 Elasticsearch + IK 分词器对知识库文档进行索引和向量检索，
   集成豆包 Embedding 模型进行文本到向量的转换，支持 2048 维向量；
   结合 KNN 向量召回 + BM25 重排序，实现"关键词+语义"双引擎搜索，
   检索准确率较关键词搜索提升 40%+
5. 基于 Kafka 解耦文件上传、处理与向量化流程，
   500M 文件上传仅需 200 毫秒
6. 基于 WebSocket + DeepSeek Stream API 实现"打字机式"流式响应，
   首响应时间从 2.3s 优化至 0.6s
7. 采用 JWT + Redis 双令牌机制，结合 ThreadLocal 管理用户上下文
8. 利用 Spring Security 实现基于组织标签的多级 RBAC 权限模型，
   从用户角色、组织归属和文件属性多维度进行权限过滤

【技术亮点】
✦ 父文档-子切片二级索引：用小子切片做精确检索（向量库），
  用大父文档做推理（文档库），通过元数据 ID 回溯，
  既保证检索精度又保留完整上下文
✦ 滑动窗口 Chunk 切分：相邻 chunk 之间保持重叠区域，
  保证跨 chunk 的信息完整性
✦ 服务降级与容错：豆包向量 API 失败时自动回退到纯文本搜索，
  Reactor 重试机制支持固定延迟重试 3 次，30 秒超时保护
✦ 会话记忆管理：每用户唯一会话 ID，对话内容时序存储，
  设置 7 天过期，保证多轮上下文完整性
```

### 🎯 这份简历的厉害之处（你做项目时要学的）

1. **每条都讲"做了什么 + 用什么技术 + 效果数字"** —— STAR 法则的工程师版
2. **量化指标极其密集** —— 简历里 9 个数字，每个都能成为面试谈话的话题
3. **技术深度可见** —— 不是"用了 Redis"，是"用 Redis BitMap"（**面试官会眼睛一亮**）
4. **业务价值清晰** —— 每个亮点都答了"为什么这么做"

## 二、量化指标速查表（项目时往这些方向凑）

### 📊 RAG / 知识库类指标

| 维度 | 范本数据 | 测量方式 |
|------|---------|---------|

| 文件处理 | 500M / 200ms | 同上 |
| 接口响应 | 800ms → 200ms | 同上 + JMeter 压测 |
| 检索准确率 | +40% (vs 关键词) | 准备 100 个真实问答对，算 Top-5 命中率 |
| 流式首响应 | 2.3s → 0.6s | 从发请求到第一个 token 的时间 |

| 文档规模 | 50 万条 / 检索 < 300ms | 数据量 + QPS |
| 幻觉率 | 6.2% → 0.7% | 100 题人工标注 |
| 召回率 | 提升至 83%+ | Top-K 命中率 |
| 内存优化 | 1000 分片 / 125 字节 | BitMap 优势点 |

### 📊 多模型 / 成本优化类

| 维度 | 范本数据 |
|------|---------|
| 推理成本 | $0.18 → $0.024 / 文档 |
| 年节省成本 | $1.8M（自托管 + 量化 + 投机解码）|

| QPS 提升 | 50 → 500（缓存 + 批量）|

### 💡 量化指标的"埋点"原则

写代码时**主动留这 4 类记录**：

```java
// 1. 关键路径的耗时
long start = System.currentTimeMillis();
// ... 业务代码 ...
log.info("[METRIC] embeddingTime={}ms", System.currentTimeMillis() - start);

// 2. 命中率（缓存、检索）
log.info("[METRIC] cacheHitRate={}/{}", hits, total);

// 3. 重要节点的成功率
log.info("[METRIC] toolCallSuccess={}", success);

// 4. 资源使用（内存、token）
log.info("[METRIC] tokenUsed={}, costYuan={}", tokens, cost);
```

**所有 `[METRIC]` 日志攒到一起，就是简历素材**。

## 三、资深级（Staff）项目描述对照（你 7 年经验对标这个级别）

### 📌 英文样本（Staff AI Engineer @ YC 公司）

```
Staff AI Engineer @ Foundry.dev (YC W24)，2024.2 - 至今

• 构建多 Agent 编排器（LangGraph + Temporal），
  支撑 Coding Agent 产品 → 48K 周活用户，任务完成率中位 71%

• 设计评估管道：12 个真实任务套件、4,800 条评分轨迹，
  对候选 Prompt 与模型进行夜间 A/B 测试

• 通过 vLLM 自托管部署（Llama 3.3 70B）+ 量化 + 投机解码，
  年节省推理成本 $1.8M

• 撰写内部 MCP server 规范，2 个月内 6 个团队采用，
  替代 14 个定制工具集成

• 主导 Autonomous Agent 安全评审，落地用量上限/限流/Kill-switch，
  预防了 2 起客户事故
```

### 🔥 资深级写法的 5 个特征

| 特征 | 普通工程师 | 资深工程师 |
|------|----------|-----------|
| **量化** | 没数据 | 每条都有数字 |
| **范围** | "实现了功能" | "支撑 N 用户 / 节省 $M" |
| **架构** | "用了 XX" | "为什么用 XX + 取舍" |
| **横向** | 一个项目 | 跨团队推动 / 写规范 |
| **风险** | 不提 | 主动提"做了什么风险预案" |

### 💡 你现在的 7 年 Java 经历，套这套写法的方法

你过去做过的事情，**先用资深级模板重写**：

```
❌ 旧版："基于 Spring Boot 实现了订单系统，QPS 达到 5000"

✅ 新版：
  • 主导订单系统重构，从单体演进到 SpringCloud 微服务，
    QPS 由 800 → 5000，订单超时率由 2.3% → 0.1%

  • 设计基于 Redis + 分布式锁 + 消息补偿的最终一致性方案，
    替换原有的两阶段提交，吞吐提升 3 倍，
    线上零数据不一致事故运行 14 个月

  • 撰写《订单中台接入规范 v2》，3 个月内 4 个上游业务系统接入，
    沉淀复用率达 80%
```

## 四、技能栈怎么写（避免堆热词）

### ❌ 烂大街写法（HR 看了直接划走）

```
熟悉大模型
熟悉 RAG
熟悉 LangChain
熟悉 Prompt 工程
熟悉向量数据库
```

**问题**：每个人都这么写，看不出你和别人的差别。

### ✅ 分层写法（一眼看出结构 + 深度）

```
编程语言    ▎Java 17 (精通,7 年) | Python (熟练,1 年) | SQL
后端框架    ▎Spring Boot 3 / Spring AI / Spring WebFlux / Spring Security / Spring Cloud
AI 工程化   ▎大模型应用开发、RAG 架构设计、Prompt 工程、Function Calling、Agent 编排
向量 / 检索 ▎Milvus / pgvector / Elasticsearch (KNN) / 混合检索 (BM25 + Embedding)
模型生态    ▎OpenAI / Claude / 通义千问 / DeepSeek / Ollama 本地部署
中间件      ▎MySQL / Redis / Kafka / Elasticsearch / MinIO
工程能力    ▎多租户架构、流式推理 (SSE/WebSocket)、限流熔断、链路追踪、压测调优
```

### 🎯 4 个关键设计

1. **分层** —— 让人一眼看到你的能力图谱（语言 / 框架 / 业务方向 / 中间件）
2. **标深度** —— 在重点项目后加 `(精通,7年)` `(熟练,1年)`，**坦诚 + 显得清晰**
3. **重要分类放前面** —— 你的核心是"Java + Spring AI"，所以 Java 和 Spring 排第 1、2 行
4. **AI 部分用"做的事"而不是"工具名"** ——
   写"RAG 架构设计"比"LangChain"强；写"Agent 编排"比"AutoGen"强

### 🚫 不要写的东西

- ❌ "熟悉计算机网络 / 数据结构" —— 7 年了，写这些显得初级
- ❌ "了解 Hadoop / Spark / 大数据" —— 不是你的方向，淡化
- ❌ "熟悉 Vue / React" —— 除非你强调全栈，否则别让面试官以为你想做前端
- ❌ "了解 Docker / Linux" —— 这是基础线，写"熟悉 K8s + DevOps 流水线"才有分量

## 六、最后：给你的 3 条最实用的建议

### 1. 你的 7 年 Java 是核心资产，**别在简历里弱化**

很多人转型时拼命堆 AI 关键词，反而淡化了优势。你的写法应该是：

> "8 年 Java 后端 + 1 年 AI 应用" —— 让人看了想：**这是带着架构能力来做 AI 的资深工程师**，
> 不是"刚学了 LangChain 的新人"。

### 2. 阶段 3-4 做项目时，**主动埋量化指标**

派聪明那份简历的"威慑力"全在数字。你做项目时要主动记录：
- 文件上传 / 检索 / 推理的耗时（前后对比）
- 准确率 / 召回率 / 幻觉率
- QPS / 并发数 / 内存占用
- 成本节省（API 调用次数减少）

**写代码时就 `log.info("[METRIC] ...")` 打出来，简历就是这么一句一句攒出来的**。

### 3. 阶段 5 时按真实项目反推简历

现在还没必要完整写简历——你**还没做项目**呢。但**这份"对标参考"是做项目时的灯塔**：

> "派聪明的 1GB → 3s，我也要做出类似数字"
> "Staff Engineer 那条 vLLM + 量化，我可以试试 Ollama 本地部署"
> "RBAC 权限我做过，可以直接拿过来"
