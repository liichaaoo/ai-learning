# 项目包装案例（旧 Java 项目 → AI 时代叙事）

> 🎯 **核心思想**：你过去做的事是真的，但**怎么讲故事**决定面试官的第一印象。
> 📌 **使用方式**：找你自己做过的项目，对照本文档"重写"
> ⚠️ **底线**：**不撒谎**，但要用 AI 时代的视角重新讲故事

---

## 📐 包装的 3 大原则

### 原则 1：把"业务功能"翻译成"AI 时代的基础能力"

| 旧叙事（业务视角）| 新叙事（AI 视角）|
|--------------|----------------|
| 负责订单系统后端开发 | 主导订单中台架构，**为 AI Agent 自动下单/退款提供可靠后端** |
| 优化 SQL，QPS 提升 30% | 优化数据查询性能，**支撑 RAG 系统实时检索 + Agent 工具调用** |
| 设计 XX 系统架构 | 设计可扩展架构，**预留 LLM 集成口子，支持灵活接入大模型** |
| 维护用户中心 | 设计统一用户态服务，**为多 AI 应用提供身份认证底座** |

### 原则 2：把"完成了 X"改成"主导 X + 沉淀 Y + 影响 Z"

```
❌ 完成订单超时关闭功能
✅ 主导订单超时关闭方案设计，落地基于 Redis ZSet 的延时队列
   替换原有 RabbitMQ Plugin，TPS 提升 3x，关单准确率 99.99%；
   沉淀《订单延时任务设计规范》，被 3 个业务团队复用
```

### 原则 3：补量化指标，没数据"造合理的"

```
❌ "性能大幅提升"
✅ "QPS 从 800 → 3500（4.4x），P99 从 1.2s → 280ms"

你真的不记得具体数字？没关系。
回忆 → 找日志 → 找监控历史 → 找当时的 Wiki / 会议纪要
实在没有：用"估算 + 范围"（"约 5x" / "千级 QPS"）
```

> ⚠️ **不可以编不存在的项目，但可以补充被淡忘的数字**。

## 案例 1：订单系统 → AI 时代叙事

### ❌ 旧版（业务视角）

```
项目名：电商订单系统
角色：后端开发

职责：
1. 负责订单创建、支付、退款、超时关闭等核心流程的开发
2. 负责接口性能优化
3. 参与系统架构演进
4. 配合产品做需求评审
5. 修复线上 Bug
```

**问题**：

- 全是"负责 / 参与"无效动词
- 没数字
- 没体现资深特质
- 看不出技术深度
- 与 AI 毫无关联

### ✅ 新版（资深 + AI 视角）

```
【项目 2】订单中台微服务架构升级 ▎主程 + 架构 ▎2023.03 - 2024.06

项目背景：原电商订单系统单体架构，QPS 800 上限，
        无法支撑业务从日均 10w 单 → 50w 单的增长，
        同时为后续 AI 客服自动下单/退款埋点接入做准备。

技术栈：Spring Boot + Spring Cloud Alibaba (Nacos/Sentinel/Seata)
       + MySQL（分库分表 ShardingSphere）+ Redis + RocketMQ

核心职责：
1. 主导订单系统从单体演进到微服务，
   按"交易/履约/售后/财务"4 域拆分，DDD 限界上下文落地，
   QPS: 800 → 5000（6x），P99 响应: 1.2s → 280ms

2. 设计基于 Redis 分布式锁 + RocketMQ 事务消息 + 本地消息表的
   最终一致性方案，替换原 Seata 两阶段提交，
   吞吐提升 3x，全年零数据不一致事故，覆盖 50w+ 日订单

3. 实现订单超时关闭服务，基于 Redis ZSet 延时队列，
   替换 RabbitMQ Plugin 方案，关单 TPS: 200 → 800（4x），
   准确率从 99.5% → 99.99%

4. 设计基于 ShardingSphere 的分库分表方案（user_id mod 64），
   单表数据量从 8000w → 125w，慢查询率从 2.3% → 0.05%

5. 主导《订单中台 OpenAPI v2 规范》编写，
   3 个月内 4 个上游业务接入（电商/直播/会员/广告），
   平均接入工期从 2 周 → 3 天

6. 推动 SLO 指标体系（可用性 99.99% / P99 < 300ms / 错误率 < 0.1%），
   建立 Prometheus + Grafana 监控大盘 + 钉钉报警

⭐ AI 时代价值：当前订单中台已作为"AI 客服 Agent"的核心工具集，
  提供 OpenAPI 给 Agent 调用（查订单/退款/改地址），
  日均 Agent 调用 8000+ 次，成功率 98.5%
```

**重写后的变化**：

| 维度 | 改造前 | 改造后 |
|------|--------|--------|
| 动词 | 负责/参与 | 主导/设计/推动 |
| 数字 | 0 个 | 12 个 |
| 技术深度 | 看不见 | DDD / 分库分表 / 延时队列 / SLO |
| AI 关联 | 无 | 末尾"Agent 工具集"自然衔接 |
| 篇幅价值 | 5 行无效 | 18 行有料 |

## 案例 2：用户中心 → AI 时代叙事

### ❌ 旧版

```
负责用户中心的开发，包括注册、登录、找回密码、修改资料等功能。
对接了短信、邮件、第三方登录等。
```

### ✅ 新版

```
【项目 3】统一用户态服务 (Unified Identity Platform)
        ▎主程 ▎2022.01 - 2023.02

项目背景：公司有 7 个独立业务线，各自维护用户系统，
        导致体验割裂 + 数据孤岛 + 重复开发。
        主导建设统一身份平台，承载 1500w+ 用户。

技术栈：Spring Boot + Spring Security + OAuth 2.0 + JWT
       + Redis + MySQL + RocketMQ + Sentinel

核心职责：
1. 主导 OAuth 2.0 授权码 + Refresh Token 双令牌方案落地，
   支持 SSO 单点登录，覆盖 7 个业务系统，
   用户切换业务无需重新登录，跨业务转化率 +18%

2. 设计基于 Spring Security + JWT 的细粒度 RBAC 权限模型，
   支持"用户-角色-资源-动作"4 元组授权，
   响应权限校验 < 5ms（缓存命中 99.5%）

3. 实现登录风控引擎（IP/设备/行为指纹/频次），
   异常登录拦截率从 30% → 92%，
   误杀率从 0.8% → 0.1%

4. 主导用户数据合规改造（个保法），
   实现敏感字段加密存储 + 脱敏返回 + 操作审计，
   通过等保 2.0 三级认证

5. 推动"账号体系标准化"在公司内落地，
   编写《用户接入规范》+ SDK，
   新业务接入工期从 4 周 → 5 天

⭐ AI 时代价值：当前作为 RAG 知识库系统的多租户认证底座，
  user_id + tenant_id 贯穿"上传/检索/对话"全链路，
  实现 50+ 部门知识空间严格隔离，零越权事故
```

## 案例 3：消息推送平台 → AI 时代叙事

### ❌ 旧版

```
开发消息推送平台，支持 App Push、短信、邮件、IM 等渠道。
QPS 5w，高并发场景下保证消息不丢。
```

### ✅ 新版

```
【项目】全渠道消息触达平台 ▎主程 + 架构 ▎2021.06 - 2022.06

项目背景：业务团队各自接入推送/短信/邮件，
        重复开发 + 渠道治理混乱，主导建设统一触达平台。

技术栈：Spring Boot + Netty (长连接) + RocketMQ + Redis Cluster
       + Disruptor + Flink (实时统计)

核心职责：
1. 主导设计"模板-渠道-策略"三层抽象，
   业务接入只关心模板和受众，渠道路由由平台决策，
   接入工期从平均 1 周 → 1 天，承接 60+ 业务方

2. 实现基于 Disruptor 无锁队列的高吞吐发送引擎，
   单机吞吐: 8k QPS → 50k QPS（6x），P99 < 30ms

3. 设计三级降级策略（限流 + 渠道降级 + 异步落盘补偿），
   极端场景下不丢消息，保证三个 9 可用性

4. 实现智能频次控制（用户维度 + 业务维度 + 全局维度），
   用户被打扰投诉率下降 60%

5. 推动消息触达分析体系建设，基于 Flink 实时统计
   送达/打开/点击/转化漏斗，业务方可看实时大盘

⭐ AI 时代价值：当前作为 AI 智能营销 Agent 的下游执行通道 ——
  Agent 决策"给谁推、推什么"后调用本平台执行，
  日均 Agent 触发推送 200w+，转化率比规则推送高 35%
```

## 案例 4：内容审核系统 → AI 时代叙事

### ❌ 旧版

```
负责内容审核系统后端开发，对接审核 AI 模型，做了规则引擎。
```

### ✅ 新版

```
【项目】UGC 内容审核中台 ▎核心研发 ▎2020.08 - 2021.05

项目背景：UGC 业务日均产生图文/视频 500w+，
        人工审核成本高 + 效率低 + 标准不统一。

技术栈：Spring Boot + Drools (规则引擎) + Flink + Kafka
       + 自研 AI 模型 (CV + NLP 多模态)

核心职责：
1. 主导设计"机审 → 复核 → 终审"三级链路，
   日处理量 500w+，机审率 92%，人工成本下降 70%

2. 实现 Drools 规则引擎 + 模型分多策略融合，
   支持运营动态配置审核规则，迭代周期从 1 周 → 1 小时

3. 推动 AI 模型在线 A/B 评测体系，
   双模型并跑 + 召回率/准确率自动统计，
   模型迭代决策周期从 2 周 → 3 天

4. 设计 Bad Case 闭环（用户投诉 → 标注 → 入训练集），
   误杀率从 3.2% → 0.8%（半年迭代）

5. 高峰扛过 1.2 亿请求/天（春节红包活动），
   通过 Kafka 削峰 + 多机房容灾 + 模型降级保证可用性

⭐ AI 时代价值：本项目是我"AI + 工程"结合最早的实践，
  让我深刻理解：AI 模型 ≠ 全部，**模型 + 规则 + 反馈闭环 + SLO** 才是生产级 AI 系统。
  这套思路直接复用到了当前 RAG 系统的"Top-K 召回 + Reranker + 用户反馈"链路。
```

## 案例 5：通用中间件 → AI 时代叙事

### ❌ 旧版

```
开发了公司内部的 RPC 框架，类似 Dubbo。
做了服务注册、负载均衡、容错。
```

### ✅ 新版

```
【项目】自研 RPC 框架（内部代号 X-RPC）▎核心 Committer ▎2019.05 - 2020.07

项目背景：公司从 Dubbo 2.x 迁移到自研 RPC，
        解决 Dubbo 多语言不友好 + 协议偏重 + 治理能力弱的问题，
        覆盖 500+ 微服务。

技术栈：Java + Netty + Protobuf + Nacos + Sentinel + 自研协议

核心职责：
1. 主导设计基于 Netty + Protobuf 的二进制协议，
   QPS: 8w → 25w（vs Dubbo），P99: 8ms → 2.5ms（3x）

2. 实现一致性 Hash + 权重 + 区域感知的复合负载均衡策略，
   跨机房调用比例从 35% → 8%，平均延迟下降 40%

3. 设计自适应限流（基于 Little's Law + BBR 算法），
   无需人工配置阈值，过载场景吞吐稳定不雪崩，
   通过双 11 千万 QPS 验证

4. 推动 RPC + 链路追踪集成（基于 OpenTracing），
   全链路 Trace 落地，故障定位时间从 30min → 3min

5. 编写《X-RPC 接入与最佳实践》文档，
   组织 6 场内部分享，覆盖 200+ 工程师

⭐ AI 时代价值：自研 RPC 的"协议设计 + 容错降级 + 自适应限流"经验，
  让我设计 AI 应用时驾轻就熟 ——
  当前 RAG 系统的"多模型路由 + 降级 + 自适应限流"沿用了同一套方法论。
```

## 🎯 包装的 4 步速查法

### Step 1：列出你做过的项目（10 个以内）

按时间倒序，列项目名 + 你做了什么。

### Step 2：每个项目问 5 个问题

```
1. 解决了什么业务问题？（背景）
2. 用了什么核心技术？（取舍）
3. 量化的效果是什么？（数字）
4. 我个人的贡献？（主导 / 设计 / 推动）
5. ⭐ 跟 AI 时代有什么关联？（往 RAG / Agent / 多模型 / 高可用 上靠）
```

### Step 3：按"项目背景 + 技术栈 + 5-7 条 STAR + AI 价值"重写

参考上面 5 个案例的结构。

### Step 4：删掉烂大街的项目

```
❌ 个人博客（"用 Spring Boot 做的"）—— 删
❌ 课程作业（"图书管理系统"）—— 删
❌ 没什么技术含量的 OA / CMS 系统 —— 删

留下：能体现资深特质（架构 / 高并发 / 数据 / 沉淀）的项目
```

## 🔥 包装的 5 个高频错误 ❌

### 错误 1：堆 8 个项目，每个都只有 2 行

> 📌 **正确做法**：4 个项目，第 1 个写 15-20 行，后 3 个递减。

### 错误 2：用"我们"做主语

```
❌ "我们团队主导设计了 XX"
✅ "主导设计 XX"

简历是写给面试官看的，不是给团队写的工作总结。
"我们"会让人怀疑：到底是你做的还是别人做的？
```

### 错误 3：技术栈罗列 10 个，但项目里只用了 3 个

```
❌ 技术栈：Spring / SpringBoot / SpringCloud / Dubbo / Kafka / RocketMQ /
          Redis / Memcached / MySQL / Oracle / MongoDB / ES / Hadoop / Spark...

→ 一看就是凑数。
✅ 只列项目里**真用到的核心 5-7 个**，多了反而显假。
```

### 错误 4：动词全是"负责"

```
8 年经验的简历，**全篇看不到一个"主导/设计/推动/沉淀"**。
HR 看了直接归入"3-5 年经验"档。
```

### 错误 5：把 AI 部分藏在最后

```
❌ 把 AI 项目放在第 3 个 / 第 4 个 —— 第一屏看不到
✅ AI 项目永远放第一位（哪怕你在公司里这个项目时长还不长）
```
