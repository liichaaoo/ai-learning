## 一、核心叙事策略

### 你的差异化定位（一句话）

> **"我是 Java + Go 双栈工程师：业务和 RAG 用 Spring AI 写，但 LLM 网关、Agent 协议层、向量召回 sidecar 这些高并发组件用 Go 写——这种混合栈是大厂内部 AI 平台的主流做法。"**

这一句话的杀伤力：
- 立刻把你和"只会 Java"的候选人拉开差距
- 立刻把你和"只会 Python"的候选人拉开差距
- 体现"按合适的工具选合适的语言"的资深判断力
- 暗示你能 cover **AI 平台 / Infra 岗位**，不止应用层

### 简历布局建议

```
┌────────────────────────────────────────┐
│ 顶部：Java + AI 主线（阶段 3/4 项目）    │ ← 70% 篇幅
├────────────────────────────────────────┤
│ 中段：Go AI 视角项目（1-2 个）           │ ← 20% 篇幅
├────────────────────────────────────────┤
│ 底部：技能栈（Java/Go/Python 都列）       │ ← 10% 篇幅
└────────────────────────────────────────┘
```

**关键词需要出现的（Go 部分）**：
- LLM Gateway / 多模型路由
- 流式响应 / SSE / Streaming
- 向量检索 sidecar
- MCP / Agent 协议
- 高并发 / goroutine / 单二进制部署
- Kafka / 流处理 / Token 计费

## 二、改写模板（5 种典型 Go 业务）

> 每个模板包含：❌ 旧叙事 → ✅ AI 视角叙事 → 💡 面试可被追问的延伸点

### 模板 1：高并发 API 网关 / 接入层

#### ❌ 旧叙事

> 负责 XX 网关服务的 Go 语言开发，承载 5w+ QPS，实现限流熔断、灰度发布、协议转换。

#### ✅ AI 视角叙事

> 主导 **Go 高并发网关服务**，承载 5w+ QPS。具备 **AI 时代可演进性**——在 LLM 应用普及背景下，该网关可平滑升级为 **多模型路由 LLM Gateway**：
>
> - **流式协议天然支持**：网关已具备 SSE / 长连接能力，直接对接 OpenAI 兼容协议（`/v1/chat/completions`）
> - **多维限流复用**：原有按 user/tenant/IP 限流策略，可直接扩展为按 **Token 数 / 模型类型** 限流
> - **熔断降级可复用**：原有 4 级降级链路（主备/缓存/默认值/拒绝），可演进为 **LLM 4 级降级**（贵模型→便宜模型→缓存→兜底回复）
> - **单进程 30MB 内存扛 5w QPS**，比 Java 同类网关节省 60%+ 资源

#### 💡 可被追问的延伸点

- "为什么不用 Java 写？" → 单二进制部署、内存低、冷启动快，AI 网关流量峰谷大，Go 在弹性伸缩时优势明显
- "和 one-api 区别？" → 提你看过 one-api 源码，自研版本主要强化了 \<X 能力\>
- "Token 限流怎么做？" → 用 Redis + Lua 脚本，按 prompt_tokens 预扣，response_tokens 实扣，差额回补

> 📎 直接对应系统设计题：[`../系统设计/02-多模型路由网关.md`](../系统设计/02-多模型路由网关.md)

### 模板 2：消息中间件 / Kafka 流处理

#### ❌ 旧叙事

> 用 Go 实现日志采集和 Kafka 消费链路，单实例 10w+ TPS，写入 ClickHouse 做实时分析。

#### ✅ AI 视角叙事

> 设计 **Go 高吞吐流处理链路**，单实例 10w+ TPS，可直接复用为 **LLM 应用 Token 计费/审计链路**：
>
> - **三层架构对标行业**：Kafka（接入）→ Flink-style 聚合（Go worker pool 实现）→ ClickHouse（OLAP），契合 LLM 调用产生的"高写入、按租户聚合、按时间窗口分析"特征
> - **每条事件 < 1ms 延迟**，支撑实时计费看板和异常调用告警（如某用户 1 分钟内消耗 100w token 触发熔断）
> - **Exactly-Once 语义**：通过 Kafka 事务 + ClickHouse 幂等写入实现，保证计费不重不漏

#### 💡 可被追问的延伸点

- "为什么不用 Flink？" → 团队规模小，Flink 运维成本高；Go worker pool + 状态本地化对中小规模够用
- "ClickHouse 表怎么设计？" → 按 (tenant_id, day) 分区，按 model_id 排序键，物化视图预聚合
- "怎么保证不丢消息？" → consumer 手动 commit + DLQ + 监控 lag

> 📎 直接对应系统设计题：[`../系统设计/04-Token计费系统.md`](../系统设计/04-Token计费系统.md)

### 模板 3：gRPC 微服务 / 协议转换

#### ❌ 旧叙事

> 用 Go 开发 gRPC 微服务，承担多协议转换（HTTP/gRPC/Thrift），日均调用 1 亿+。

#### ✅ AI 视角叙事

> 主导 **Go 多协议网关**，承担 HTTP / gRPC / Thrift 三协议互转，日均 1 亿+ 调用。在 **AI Agent 时代** 该能力可直接演进为：
>
> - **MCP Server 实现**：MCP（Model Context Protocol）协议在传输层基于 stdio / Streamable HTTP，本质上是一种新的 RPC 模式；Go 在 gRPC / 长连接 / 协议解析上的积累可零成本迁移
> - **Agent 工具网关**：把企业内部已有 RPC 服务封装成 Agent 可调用的工具集合，统一鉴权、限流、审计
> - **A2A（Agent2Agent）协议接入**：Google 推的 Agent 互联协议天然 gRPC，复用现有协议网关基建

#### 💡 可被追问的延伸点

- "MCP 和 Function Calling 区别？" → FC 是 LLM 内置能力但每个模型 schema 不同；MCP 是统一协议，工具能跨 LLM 复用
- "你的 MCP Server 怎么做鉴权？" → token-based + 工具白名单，按调用方分级
- "怎么防止 Prompt Injection 通过工具调用攻击？" → 工具入参强 schema 校验、沙箱执行、危险操作二次确认

> 📎 配套深读：[`../../04-Agent与工程化/Week2-Agent设计模式/Day4-MCP协议.md`](../../04-Agent与工程化/Week2-Agent设计模式/Day4-MCP协议.md)

### 模板 4：缓存/存储中间件

#### ❌ 旧叙事

> 用 Go 实现分布式缓存代理，多级缓存（local + Redis + DB），命中率 95%+。

#### ✅ AI 视角叙事

> 设计 **Go 多级缓存代理**，本地 + Redis + DB 三级，命中率 95%+。该架构是 **LLM 应用降本增效的核心基建**：
>
> - **LLM 调用三层缓存**：完全相同 prompt 走精确缓存（key=hash(prompt+params)）→ 语义近似 prompt 走向量缓存（embedding 相似度 > 0.95）→ 最后才打模型
> - **预估场景命中率 30-50%**：客服 / FAQ 类场景重复 prompt 极多，可节省 50% 模型调用成本
> - **失效策略 LLM 化**：常规缓存按 TTL，LLM 缓存按"内容时效性 + 模型版本"双维度，模型升级自动失效

#### 💡 可被追问的延伸点

- "向量缓存怎么实现？" → Redis + RediSearch 向量索引，或独立 Milvus 集群
- "怎么避免缓存污染？" → 写入前打分，低质量回答（用户负反馈、长度异常）不入缓存
- "和 GPTCache 区别？" → GPTCache 是 Python 库且偏单机；自研版面向多租户、和现有缓存治理体系打通

### 模板 5：搜索/召回服务

#### ❌ 旧叙事

> 用 Go 开发搜索召回服务，对接 ES，实现倒排 + 多路召回，P99 < 50ms。

#### ✅ AI 视角叙事

> 主导 **Go 搜索召回服务**，多路召回 P99 < 50ms。在 **RAG 时代** 该服务可平滑演进为 **混合检索引擎**：
>
> - **从 BM25 到向量混合检索**：保留原有 ES BM25 关键词召回，新增 Milvus 向量召回，**RRF（Reciprocal Rank Fusion）融合**两路结果
> - **作为 Java RAG 服务的 sidecar 部署**：Java 业务专注编排和 LLM 调用，Go sidecar 专注高 QPS 召回，**单 sidecar 1w QPS、内存 < 200MB**
> - **召回质量提升 20%+**：纯 BM25 在长尾 query 命中差，向量召回补全语义相关；混合后 NDCG@10 从 0.62 提升到 0.78

#### 💡 可被追问的延伸点

- "为什么不直接 Java 一把梭？" → 召回是高 QPS 路径，Go sidecar 资源占用低；Java 服务专注业务逻辑职责更清晰
- "RRF 怎么调？" → k 值默认 60，按业务 A/B 测试；可加 query 类型分流（短 query 偏 BM25，长 query 偏向量）
- "怎么处理向量库一致性？" → CDC 同步 + 双写降级，向量库故障时回退纯 BM25

> 📎 直接对应主项目：[`../../03-SpringAI与RAG/`](../../03-SpringAI与RAG/) 阶段 3 RAG 系统

## 四、面试现场话术（3 段式）

### 30 秒电梯演讲

> "我有 7 年 Java 经验做企业级应用，最近 1-2 年同时在做 Go 高并发中间件——做过 \<网关 / Kafka 链路 / 召回服务\>。我把这两块结合起来在做 AI 应用：业务和 RAG 用 Spring AI 写，LLM 网关、向量召回 sidecar、Token 计费链路这些用 Go 写。这种混合栈是大厂内部 AI 平台的主流做法。"

### 2 分钟项目讲解

> "我主项目是 \<阶段 3 RAG\>，用 Spring AI 做的，承载企业知识库 ……（按主项目讲）。
>
> 同时我把以前做的 Go \<XX 网关/服务\> 升级成了 LLM 应用基建——具体来说，原来的 \<5w QPS 网关\> 加了 OpenAI 兼容协议层，按 Token 限流、按模型路由，实测单进程 30MB 内存扛住整个团队的 LLM 调用，省了 60% 资源成本。这部分我可以展开讲。"

### 反问环节加分

> "我注意到很多 AI 平台的网关 / 计费 / 召回都是 Go 写的，请问贵司在 AI 平台这一层是 Java 一把梭，还是会按职责分语言栈？"

→ 这种反问会让面试官立刻把你定位为"有平台思维的资深工程师"。

## 五、避坑清单

### ❌ 不要这样写

- ❌ "Go 比 Java 好" → 显得不成熟，资深的人不站队
- ❌ "我精通 Go AI 框架" → 如果只看过 Eino README，别说精通
- ❌ "用 Eino 做了 RAG" → 除非你真做了完整项目，否则不要主线讲 Eino
- ❌ "Java 不适合做 AI" → 你主项目就是 Java，自相矛盾

### ✅ 要这样写

- ✅ "按职责分层，混合栈"
- ✅ 具体数字（QPS / 内存 / 成本）
- ✅ 提及行业标杆（one-api / Milvus / Ollama 都是 Go 写的）
- ✅ 给出"演进路径"而不是"已经做好"（更可信）
