# 系统设计 · 02 多模型路由网关

> 🎯 **题目**：设计一个 LLM 网关，统一接入 GPT-4 / Claude / 通义 / DeepSeek / Ollama 等多模型，按场景路由 + 按成本路由 + 限流降级 + 统一日志
>
> 💡 **类比**：这是 LLM 时代的 **API Gateway**，类似 Kong / Spring Cloud Gateway，但专为 LLM 设计

---

## 一、需求澄清

```
功能：
  - 统一 API（OpenAI 兼容协议为标准）
  - 按场景路由（代码 / 通用 / 中文 / 数学 / 多模态）
  - 按成本路由（简单问 → 便宜模型）
  - 限流（多维度：用户 / API Key / 模型）
  - 降级（主模型挂 → 备模型）
  - 流式 / 非流式都支持
  - 全链路日志（每次调用的 input/output/cost）
  - 多租户 + 配额计费

非功能：
  - 网关吞吐 ≥ 5k QPS
  - 网关本身延迟 < 50ms（不能成为瓶颈）
  - 99.99% 可用性
  - 加密（不存 plaintext key）
```

---

## 二、容量估算

```
QPS：5k 峰值
模型数：8 个上游（GPT-4o / Claude 3.5 / Qwen-plus / Qwen-turbo /
                  DeepSeek-v3 / GLM-4 / Llama-3 / Ollama 本地）
每次调用 input 4k token，output 500 token
月度调用量：5k × 86400 × 30 × 0.5（平均 50%QPS）= 65亿次/月
日志存储：每条 5KB × 65亿 / 30 ≈ 1TB/天 → 走 ClickHouse + 7 天滚动
```

### 4.1 OpenAI 兼容协议（核心抽象）

>
> 1. 业界事实标准（90% 的 SDK / 工具默认）
> 2. 客户端零改造（直接改 base_url）
> 3. 所有非 OpenAI 模型，都要做协议转换

### 4.2 路由策略 ⭐⭐ 核心

**4 种路由策略组合使用**：

```java
public class RoutingEngine {
    public Model route(Request req) {
        // 1. 显式指定（最高优先级）
        if (req.model() != null && req.model().startsWith("@")) {
            return Model.of(req.model().substring(1));  // "@gpt-4o" 强制
        }
        
        // 2. 场景路由
        Scene scene = sceneClassifier.classify(req);
        if (scene == Scene.CODE) return Model.DEEPSEEK_V3;       // 代码强
        if (scene == Scene.MATH) return Model.QWEN_MATH;         // 数学强
        if (scene == Scene.IMAGE) return Model.GPT_4O;           // 多模态
        
        // 3. 成本路由（按复杂度）
        Complexity c = complexityEstimator.estimate(req);
        if (c == Complexity.SIMPLE) return Model.QWEN_TURBO;     // 0.001 元/k
        if (c == Complexity.NORMAL) return Model.QWEN_PLUS;      // 0.005 元/k
        return Model.GPT_4O;                                      // 0.03 元/k
        
        // 4. 一致性路由（同一会话用同一模型，避免风格漂移）
        if (req.sessionId() != null) {
            return sessionStickyCache.get(req.sessionId(), this::route);
        }
    }
}
```

**复杂度估算**：

```java
public Complexity estimate(Request req) {
    int tokens = countTokens(req);
    boolean hasCode = containsCodePattern(req);
    boolean hasMath = containsMathPattern(req);
    boolean isShort = tokens < 100;
    
    if (isShort && !hasCode && !hasMath) return SIMPLE;
    if (hasCode || hasMath || tokens > 2000) return COMPLEX;
    return NORMAL;
}
```

### 4.3 限流：多维度令牌桶 ⭐

```java
// 4 个维度的令牌桶
public class MultiDimRateLimiter {
    private RateLimiter perUserLimiter;       // 单用户：60 req/min
    private RateLimiter perTenantLimiter;     // 单租户：1000 req/min
    private RateLimiter perModelLimiter;      // 单模型：1w req/min（防上游 429）
    private RateLimiter globalLimiter;         // 全局：5w req/min

    public void acquire(Request req) {
        if (!perUserLimiter.tryAcquire(req.userId())) throw new RateLimit("user");
        if (!perTenantLimiter.tryAcquire(req.tenantId())) throw new RateLimit("tenant");
        if (!perModelLimiter.tryAcquire(req.model())) throw new RateLimit("model");
        if (!globalLimiter.tryAcquire()) throw new RateLimit("global");
    }
}
```

**Redis 实现令牌桶**（Lua 脚本保证原子性）：

```lua
-- 简化版令牌桶 Lua
local tokens_key = KEYS[1]
local timestamp_key = KEYS[2]
local rate = tonumber(ARGV[1])
local capacity = tonumber(ARGV[2])
local now = tonumber(ARGV[3])
local requested = tonumber(ARGV[4])

local last_tokens = tonumber(redis.call("get", tokens_key)) or capacity
local last_refreshed = tonumber(redis.call("get", timestamp_key)) or 0
local delta = math.max(0, now - last_refreshed)
local filled = math.min(capacity, last_tokens + delta * rate)
local allowed = filled >= requested

if allowed then
  redis.call("set", tokens_key, filled - requested)
end
redis.call("set", timestamp_key, now)
return allowed and 1 or 0
```

### 4.4 降级链：CircuitBreaker

```java
@Component
public class FallbackChain {
    public ChatResponse call(Request req) {
        // 主链路
        try {
            return circuitBreaker("gpt-4o").executeSupplier(() -> gpt4oClient.call(req));
        } catch (Exception e) {
            log.warn("gpt-4o fail, fallback to qwen-plus");
        }
        
        // 降级 1
        try {
            return circuitBreaker("qwen-plus").executeSupplier(() -> qwenClient.call(req));
        } catch (Exception e) {
            log.warn("qwen-plus fail, fallback to ollama");
        }
        
        // 降级 2（本地）
        try {
            return ollamaClient.call(req);
        } catch (Exception e) {
            // 最终兜底
            return ChatResponse.template("系统繁忙，请稍后重试");
        }
    }
}
```

**熔断器配置**：

```yaml
resilience4j.circuitbreaker:
  instances:
    gpt-4o:
      failure-rate-threshold: 50      # 50% 失败率开熔断
      wait-duration-in-open-state: 30s # 半开等 30s
      sliding-window-size: 100
```

### 4.5 流式透传 SSE

```java
@PostMapping(value = "/v1/chat/completions", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
public Flux<ServerSentEvent<String>> chat(@RequestBody OpenAIRequest req) {
    Model selected = router.route(req);
    UpstreamClient client = clientRegistry.get(selected);
    
    return client.streamCall(req)
        .doOnNext(chunk -> tokenCounter.add(req, chunk))   // 旁路计费
        .doOnComplete(() -> usageRecorder.flush(req))
        .doOnError(e -> failoverLogger.log(req, e))
        .map(chunk -> ServerSentEvent.builder(chunk).build());
}
```

### 4.6 计费系统

```java
public class UsageRecorder {
    public void record(Request req, ChatResponse resp) {
        Usage u = resp.usage();
        BigDecimal cost = priceTable.compute(req.model(), u);
        
        // 1. ClickHouse 明细（异步批量写）
        kafkaProducer.send("llm-usage", new UsageRecord(
            req.tenantId(), req.userId(), req.model(),
            u.promptTokens(), u.completionTokens(), cost,
            req.timestamp()
        ));
        
        // 2. Redis 实时聚合（按租户/月扣额度）
        redis.opsForHash().increment(
            "quota:" + req.tenantId() + ":" + yearMonth(),
            req.model() + ":tokens", u.totalTokens()
        );
        
        // 3. Prometheus 监控
        meterRegistry.counter("llm.cost", "model", req.model())
            .increment(cost.doubleValue());
    }
}
```

## 五、可观测设计

```
日志：每次请求一条结构化日志，写 ClickHouse
  - request_id / tenant_id / user_id / model / latency / tokens / cost / success
  
指标：Prometheus
  - 各模型 QPS / 延迟 / 错误率 / 成本
  - 路由命中率（多少 % 走了便宜模型）
  - 熔断器状态

追踪：SkyWalking
  - 一次请求贯穿：Gateway → 路由 → 协议转换 → 上游模型

告警：
  - 单模型错误率 > 5% → 钉钉 + 自动切下一级
  - 单租户 1h 成本 > ¥1000 → 邮件预警
  - P99 > 1s → 排查
```

## 七、可能被追问

| 追问 | 关键答点 |
|------|---------|
| 怎么防 API Key 泄露？| 不存 plaintext，KMS 加密 + 客户端用临时 token |
| 路由分类器（SIMPLE/NORMAL/COMPLEX）准确吗？| 用规则 + 小模型分类（Qwen-turbo），离线评测 90%+ 准 |
| 流式怎么记录 token？| 监听上游 stream end 事件 + chunk 累计 |
| 上游模型集体挂怎么办？| 本地 Ollama + 模板回答兜底 |
| 怎么处理 prompt 缓存命中？| 网关上层做语义缓存（Redis Vector），命中直接返回 |
| Token 计费有误差吗？| 上游返回的 usage 是权威，本地估算只用于路由决策 |
