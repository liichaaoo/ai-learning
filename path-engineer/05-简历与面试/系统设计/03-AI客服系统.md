# 系统设计 · 03 AI 客服系统

> 🎯 **题目**：设计一个 AI 客服系统，目标"机器人优先回答 + 答不出转人工"
>
> 💡 **核心矛盾**：业务**怕答错**（影响客户），用户**怕没答上**（体验差），系统**怕成本高**（每月 LLM 烧钱）

---

## 一、需求澄清

```
业务场景：
  - 7×24 在线客服
  - 主流接入：网页 / App / 公众号 / 企业微信
  - 历史日均 10 万会话，机器人解决 60% 期望提升到 85%

功能需求：
  - 多轮对话 + 上下文记忆
  - RAG（FAQ + 知识库）
  - 复杂场景调 Tool（查订单/退款/改地址）
  - 答不出自动转人工 + 上下文交接
  - 用户主动喊"转人工"立即转
  - 满意度评价

非功能需求：
  - P95 首字延迟 < 1s
  - 支持 1k 并发会话
  - 99.95% 可用
  - 安全：不能泄露其他用户订单
```

---

## 二、容量估算

```
日均会话：10 万
高峰 QPS：100（白天集中）
峰值并发会话：5000
平均每会话 4 轮对话 = 40 万次 LLM 调用/天

存储：
  - 会话历史：40 万条/天 × 1KB × 90 天 ≈ 35GB
  - 知识库：5000 条 FAQ + 500 篇知识文档
  - 向量库：5000 × 1024 × 4B ≈ 20MB（小到极致）

日成本估算（不算优化）：
  40 万 × 4k token × ¥0.005/k = ¥8000/天 = ¥24w/月
  → 必须靠"FAQ 命中 + 多模型路由"压到 ¥6w/月
```

## 四、关键设计

### 4.1 意图识别（核心、决定整个体验）

```java
public Intent identify(String userInput, Context ctx) {
    // 1. 规则匹配（最快，处理 30%）
    if (matchRule("转人工|人工|找人", userInput)) return Intent.HUMAN_AGENT;
    if (matchRule("查询订单.*?\\d+", userInput)) return Intent.QUERY_ORDER;
    if (matchRule("退款|退货", userInput)) return Intent.REFUND;
    
    // 2. FAQ 向量匹配（处理 50%）
    var faqMatch = faqVectorStore.search(userInput, topK=1);
    if (faqMatch.score() > 0.92) return Intent.FAQ.with(faqMatch.id());
    
    // 3. 分类模型（处理剩余 20%，慢但准）
    return classifierLLM.classify(userInput, ctx);
}
```

**经验**：规则 + FAQ 命中能解决 60%-80% 简单问题，**省 80% LLM 成本**。

### 4.2 RAG 知识库

参考 [`./01-企业RAG系统.md`](./01-企业RAG系统.md)，但**两点定制**：

1. **领域 prompt**：

```
你是 XX 公司客服小助手。
- 只能基于"参考资料"回答
- 涉及订单/账户/支付，必须先调 Tool 查询
- 不知道的问题，回答"我先帮您转人工"
- 语气友好、专业、不超过 100 字
```

2. **拒答触发转人工**：

```java
String answer = chatClient.prompt(req)...content();
if (containsPattern(answer, "我不知道|无法回答|建议联系客服")) {
    return new ChatResponse(answer, true);  // hand_off=true，前端立即转人工
}
```

### 4.3 Function Call：业务工具集

```java
@Component
public class CustomerServiceTools {
    
    @Tool("查询订单状态。需要订单号。")
    public OrderStatus queryOrder(@ToolParam String orderId, @AuthContext UserCtx ctx) {
        // 关键安全：只能查当前用户的订单！
        Order order = orderApi.findById(orderId);
        if (!order.userId().equals(ctx.userId())) {
            throw new ForbiddenException("不是您的订单");  // 防越权
        }
        return order.status();
    }
    
    @Tool("发起退款。退款前必须先查订单状态确认可退。")
    public RefundResult refund(@ToolParam String orderId, @ToolParam String reason, @AuthContext UserCtx ctx) {
        // 同样校验
        // 写工单（人工审核），不直接打款（防 Prompt Injection 套现）
        return refundService.createTicket(orderId, reason, ctx.userId());
    }
}
```

⚠️ **安全红线**：
- **每个 Tool 都要校验 user_id**（不能信 LLM 传的参数）
- **涉及钱的 Tool 必须走人工审核**（防 Prompt Injection 损失）
- **敏感字段返回前脱敏**（手机号/身份证）

### 4.4 多轮对话 + 上下文压缩

```java
public class ConversationManager {
    public List<Message> getContext(String sessionId) {
        // 最近 3 轮原文 + 早期摘要 + 用户档案
        var recent = redis.lrange("session:" + sessionId, -6, -1);  // 3 轮 = 6 条
        var summary = redis.get("session:summary:" + sessionId);
        var profile = userProfileService.get(sessionId);
        
        return List.of(
            new SystemMessage(systemPrompt() + "\n用户档案：" + profile),
            new SystemMessage("早期对话摘要：" + summary),
            ...recent
        );
    }
    
    // 后台异步：每 10 轮压缩一次
    @Scheduled
    public void compress() {
        var oldMessages = redis.lrange("session:xx", 0, -10);
        var summary = chatClient.prompt("总结以下对话核心信息（200 字内）：" + oldMessages).call().content();
        redis.set("session:summary:" + sessionId, summary);
        redis.ltrim("session:xx", -10, -1);  // 保留最近 10 条
    }
}
```

### 4.5 转人工流程（最重要的兜底）

```
触发条件（任一满足）：
  1. 用户主动说"转人工"
  2. 用户连续 3 次回复"不满意/没解决"
  3. 模型回答含"无法回答/转人工"
  4. 涉及金额超 ¥500 的退款（强制审核）
  5. 用户情绪检测到"愤怒"（情感分析模型）
  6. 单会话超 20 轮还没解决

转人工动作：
  1. 立即停止 AI 回复
  2. 把会话历史 + 用户档案 + 已尝试动作打包成"上下文摘要"
  3. 写工单 + 路由给坐席（按技能/排队）
  4. 推送 IM 通知
  5. 坐席接入后，AI 全部记录上下文供坐席查阅
```

**上下文交接示例**：

```
【AI 客服→人工 交接摘要】
用户：138****1234（VIP 银卡，老客）
会话时长：8 分钟，对话轮次：12 轮
核心诉求：订单 #20260515123456 退款，原因"质量问题"
已尝试：
  ✓ 查询订单状态（已发货 3 天，可退）
  ✓ 创建退款工单 #RF20260517001（待审核）
  ✗ 用户对处理时长不满（"24 小时太久"）
建议：人工审批快速通道，缩短处理时长
```

### 4.6 满意度评价 + 负反馈闭环

```
用户回答后：
  ↓
1. 隐式信号：是否再追问 / 是否说"谢谢"
2. 显式评价：👍/👎 + 文本反馈
3. 负反馈进训练队列：
   - 标注答案错在哪
   - 进周度运营 review
   - 更新 FAQ 库 / 调 prompt / 加 Tool
```

## 五、监控与告警

```
业务指标：
  - 机器人解决率 = 1 - 转人工率
  - 用户满意度（👍 / 总数）
  - 平均轮次（理想 < 5）
  - 平均会话时长

技术指标：
  - 首字延迟 P95 < 1s
  - 端到端延迟 P95 < 3s
  - LLM 调用错误率 < 0.5%
  - 工单转交成功率 > 99%

告警：
  - 机器人解决率 < 60% → 业务侧告警（可能 prompt 被破坏）
  - 平均轮次 > 8 → 模型质量退化
  - 工具调用错误率 > 5% → 上游业务系统故障
```

## 六、可能被追问

| 追问 | 关键答点 |
|------|---------|
| 用户问"我去年买的 XX"，怎么处理？| 调 Tool 查历史订单（要校验用户身份）|
| 用户说"如果不退我就投诉媒体"怎么办？| 情绪检测 → 立即转人工 + 高优先级标记 |
| 模型回答"建议您去 XX 部门"，但我们没那个部门？| 评测集发现 + 加到拒答模式 + system prompt 加约束 |
| 不同 BG 共用客服系统怎么隔离？| 多租户 tenant_id + 知识库分库 + Tool 白名单分组 |
| 半夜流量低，但还要在线？| 冷服务降级到 Ollama 本地模型，省云 LLM 钱 |
| 用户用方言或拼音错别字？| 加一道 Query Rewriter（用 LLM 标准化），再走主流程 |

## 七、答题加分点

1. **意图识别 3 级路由**（规则 → FAQ → 模型分类，命中率最大化）
2. **每个 Tool 必校验 user_id**（资深安全意识）
3. **涉钱业务走工单**（防 Prompt Injection）
4. **6 种触发转人工 + 上下文交接摘要**（产品视角）
5. **情感分析 → 自动升级**（用户体验）
6. **负反馈闭环 → 周度运营**（迭代闭环）

## 八、与"企业 RAG 系统"的差异

| 维度 | RAG 知识库 | AI 客服 |
|------|-----------|---------|
| 目标 | 答得准 | 答得准 + 兜得住 + 体验好 |
| 重点 | 检索质量 | 路由 + 转人工兜底 |
| 工具 | 检索 | 检索 + 业务 API + 工单 |
| 安全 | 数据隔离 | 数据隔离 + 防越权 + 防套现 |
| 评估 | Recall@K | 解决率 + 满意度 + 轮次 |
