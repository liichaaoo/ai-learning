## 一、RAG 整体认知 ⭐⭐⭐

### Q1：RAG 是什么？为什么需要它？

**60 秒答**：

> RAG = **Retrieval-Augmented Generation**，检索增强生成。
>
> **痛点**：LLM 三大局限
>
> 1. **知识截止**：训练数据有时间边界（GPT-4o 到 2024.10）
> 2. **私有知识**：模型不知道你公司的内部文档
> 3. **幻觉**：编造不存在的事实
>
> **RAG 流程**：
>
> ```
> 用户提问
>    ↓
> Embedding 检索（向量库 + BM25）
>    ↓
> Top-K 相关片段
>    ↓
> 拼到 Prompt 里 + 用户问题
>    ↓
> LLM 基于片段生成答案 + 引用来源
> ```
>
> **本质**：把"开卷考试"的资料先给模型，让它"看着资料答题"。

### Q2：RAG 完整流程的 8 个步骤？

```
1. 文档加载（DocumentReader）：PDF/Word/MD/HTML
2. 文档清洗：去 HTML 标签 / 去无用空白 / 修复编码
3. 文档切分（TextSplitter）：按 token / 段落 / 语义切
4. Embedding（EmbeddingModel）：向量化
5. 存储（VectorStore）：写向量 + 元数据 + 原文
6. 检索（Retriever）：向量召回 + BM25 + 重排
7. 生成（ChatClient）：拼 prompt + 调模型
8. 引用（CitationAdvisor）：抽 source 返回前端
```

**前 5 步是"离线索引"，后 3 步是"在线问答"**，工程上分两条流水线。

## 二、文档处理 ⭐

### Q3：怎么选切分策略？

| 策略 | 适用 | 优点 | 缺点 |
|------|------|------|------|
| 固定长度 | 任意 | 简单 | 切断语义 |
| 按段落 | 结构化文档 | 保语义 | 段落长度不均 |
| 按 Token | 精确控成本 | 模型友好 | 跨段落 |
| **递归切分** ⭐ | 通用首选 | 保层级 + 控长度 | 实现稍复杂 |
| 按语义 | 学术/法律 | 最高质量 | 慢 + 贵 |

**Recursive Splitter 经典实现**：

```
按以下分隔符递归尝试，直到 chunk 长度 < target：
  ["\n\n", "\n", "。", "！", "？", "，", " ", ""]
```

**经验值**：
- chunk_size = **512 token**（中文约 800 字）
- overlap = **50 token**（防跨段信息丢失）

### Q4：父文档-子切片二级索引设计？⭐ 高级题

**60 秒答**：

> 难题：**chunk 大** → 上下文丰富但检索精度差；**chunk 小** → 检索准但上下文不够。
>
> **解法**：父子两级
>
> ```
> 父文档（1024 token）：用于"喂给模型生成"
>     ↓ 切分
> 子切片（256 token）：用于"向量检索精确匹配"
> ```
>
> 子切片元数据里存 `parent_id`，检索 → 拿子切片 → **回溯父文档** → 喂给模型。
>
> **效果**：检索精度（小 chunk 优势）+ 上下文质量（大 chunk 优势）兼得。

## 三、向量检索 ⭐⭐

### Q5：向量数据库怎么选？

| 维度 | Milvus | pgvector | Elasticsearch | Qdrant | Chroma |
|------|--------|----------|---------------|--------|--------|
| 性能 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐ |
| 数据量 | 10亿+ | 千万 | 亿级 | 亿级 | 百万 |
| 复合查询 | 强 | 弱 | **极强**（混合检索）| 强 | 弱 |
| 部署 | 中（Docker）| 极简（PG 插件）| 中 | 极简（单文件）| 极简 |
| 中小项目首选 | | ✅ | ✅ | | ✅ |
| 大规模生产首选 | ✅ | | | ✅ | |

**决策树**：

```
数据量 < 100w 且已用 PG → pgvector
数据量 < 100w 且已用 ES → ES KNN
数据量 1000w+ 或要专业向量库 → Milvus
团队偏小 + 单文件部署 → Qdrant / Chroma
```

### Q6：向量召回 vs BM25 vs 重排，三阶段怎么协作？

**60 秒答**：

```
Stage 1：向量召回（语义）→ Top 50
   ↓ 解决"换说法"问题：用户问"如何上传"，文档写"文件附件"也能命中
   
Stage 2：BM25 召回（关键词）→ Top 50
   ↓ 解决"专有名词"问题：人名/产品名/编号必须精确匹配
   
Stage 3：合并（RRF 算法 / 加权和）→ Top 20
   ↓ 让两路召回结果各取所长
   
Stage 4：Reranker 重排（Cross-Encoder）→ Top 5
   ↓ 用更精准但更慢的模型对前 20 重新打分
```

**为什么这样设计**：
- 召回阶段**强 Recall 弱 Precision**（多召回，别漏）
- 重排阶段**强 Precision 弱速度**（精挑细选，慢点没事）
- 双引擎互补，比单引擎平均提升 15-25% Recall

### Q7：RRF（Reciprocal Rank Fusion）公式？

```
score(d) = Σ 1 / (k + rank_i(d))

k 是常数，通常 60
rank_i(d) 是 d 在第 i 路召回结果里的排名
```

**优点**：不需要做 score 归一化（向量得分和 BM25 得分不同量级）。

## 四、RAG 调优 ⭐⭐⭐

### Q8：RAG 答非所问怎么排查？6 步法

> | 步骤 | 检查 | 判断 |
> |------|------|------|
> | 1 | 看检索结果 | 文档片段对不对 |
> | 2 | 文档对但答不对 → 模型问题 | 调 prompt |
> | 3 | 文档不对 → 检索问题 | 看 Embedding |
> | 4 | Embedding 模型选错 | 中文用 bge-m3 别用 ada-002 |
> | 5 | chunk 切分太大/太小 | 试 256 / 512 / 1024 |
> | 6 | 改用混合检索 + Reranker | 加 BM25 + Cross-Encoder |

### Q9：RAG 怎么防幻觉 + 加来源引用？

**Prompt 三道锁**：

```
你是企业知识库助手。请严格按以下规则：

【规则 1】只能基于以下"参考资料"回答，不能编造。
【规则 2】如果参考资料不足，必须回答"根据现有资料无法回答"，不得猜测。
【规则 3】每条结论后必须标注来源 [doc_id]，无来源则不输出。

【参考资料】
{context}

【用户问题】
{question}
```

**工程兜底**：

```java
// CitationAdvisor 后处理
String answer = chatClient.prompt(...).call().content();

// 1. 校验是否有 [doc_id] 引用
if (!answer.contains("[")) {
    return "根据现有资料无法回答您的问题。";
}

// 2. 抽取引用的文档作为 source 返回
List<Source> sources = extractCitations(answer, retrievedDocs);
return new RagResponse(answer, sources);
```

### Q10：top_k 选多少合适？

> **没有银弹**，但有指导：
>
> | 场景 | top_k | 原因 |
> |------|-------|------|
> | 简单问答 | 3-5 | 答案在 1-2 段里 |
> | 需要综合多文档 | 8-10 | 跨段总结 |
> | 复杂分析 | 15-20 | 但会逼近 context 上限 |
>
> **更好的做法**：动态 top_k（按相似度阈值取，比如 score > 0.7 的全要）。

## 五、Agent 核心 ⭐⭐⭐

### Q11：什么是 Agent？跟 RAG 的区别？

**60 秒答**：

> | 维度 | RAG | Agent |
> |------|-----|-------|
> | 核心 | "找资料 → 答" | "规划 + 调工具 + 多轮决策" |
> | 行为 | 一次完成 | 循环（决策 → 行动 → 观察）|
> | 工具 | 只调向量库 | 调任意 API（DB/HTTP/RPC/SQL/Shell）|
> | 输出 | 文本 | 文本 + 副作用（写库/发邮件等）|
>
> **类比**：RAG = 实习生帮你查资料；Agent = 实习生帮你"查资料 + 改代码 + 提 PR + 通知 leader"。

### Q12：ReAct 模式原理？

**60 秒答**：

> ReAct = **Reason + Act**，三段循环：
>
> ```
> Thought（思考）：分析现状，决定下一步
>     ↓
> Action（行动）：调一个 tool（如查 DB）
>     ↓
> Observation（观察）：tool 返回结果
>     ↓ （回到 Thought，直到决定 Final Answer）
> ```
>
> **经典 prompt**：
>
> ```
> 你是 ReAct Agent。每次按以下格式输出：
> Thought: 思考
> Action: tool_name
> Action Input: 参数
> Observation: (会被填入)
> ... 重复 ...
> Thought: 我已知道答案
> Final Answer: 最终回答
> ```

### Q13：ReAct 怎么防死循环？

> 4 招：
>
> 1. **最大迭代数**：默认 10 步，超了强制停
> 2. **重复检测**：相同 Action+Input 出现 2 次直接停
> 3. **token 预算**：累积超 16k 强制停
> 4. **超时**：单次会话 > 60s 停

### Q14：ReAct vs Plan-and-Execute 区别？

| 维度 | ReAct | Plan-and-Execute |
|------|-------|------------------|
| 决策时机 | 边走边想（每步独立决策）| 一次规划，依次执行 |
| 适用 | 探索性任务（不知道步骤） | 确定性任务（步骤明确）|
| 容错 | 强（错了换路）| 弱（计划错了全错）|
| Token | 多（每轮带历史）| 省（只规划一次）|
| 例子 | "帮我查一个不熟的 API" | "处理 100 个 PR" |

**生产建议**：**先 Plan-and-Execute（省钱），失败时降级 ReAct（容错）**。

### Q15：Multi-Agent 协作的坑？

> 5 个高频坑：
>
> 1. **沟通损失**：Agent A 总结给 B，B 总结给 C，**信息逐级丢失**
>    - 解：保留原文 + 摘要双写
>
> 2. **角色越界**：Reviewer 假装自己是 Coder，自己改代码
>    - 解：每个 Agent 独立 system prompt + 工具白名单
>
> 3. **死锁**：A 等 B，B 等 A
>    - 解：DAG 编排 + 不允许循环依赖
>
> 4. **Token 爆炸**：每个 Agent 都带全量历史
>    - 解：每个 Agent 只看自己需要的输入
>
> 5. **难调试**：哪个 Agent 出的错？
>    - 解：每个 Agent 独立 trace_id + 日志染色

## 六、MCP 协议 ⭐ 2026 必会

### Q16：什么是 MCP？

**60 秒答**：

> MCP = **Model Context Protocol**，Anthropic 2024.11 发布的**标准化协议**，
> 让 LLM 客户端（Claude/Cursor/Cline 等）和工具服务（数据库/Git/文件系统等）
> 通过统一协议互通，类似"AI 应用的 USB 接口"。
>
> **三个核心对象**：
>
> | 对象 | 类比 | 作用 |
> |------|------|------|
> | Tools | Function Calling | 模型可调用的动作 |
> | Resources | 文件/URL | 模型可读取的数据 |
> | Prompts | 模板 | 复用的 prompt 模板 |
>
> **三个核心动词**：list / read / call

### Q17：MCP 与 Function Calling 关系？

> **MCP = Function Calling 的标准化升级版**。
>
> | 维度 | Function Calling | MCP |
> |------|------------------|-----|
> | 层次 | 模型 API 层 | 协议层 |
> | 工具发现 | 应用代码硬编码 | 客户端自动发现 |
> | 复用 | 每项目重写 | 一次写好，所有 MCP 客户端共享 |
>
> **类比**：FC 是"公司内部 API"，MCP 是"WebService/REST 标准"。

## 七、综合应用题

### Q18：你做的 RAG 项目用到了哪些防幻觉手段？

**模板答（贴你项目）**：

> "我项目里用了 5 道防线：
>
> 1. **Prompt 硬约束**：'只基于参考资料回答，无依据必须说不知道'
> 2. **强制引用 [doc_id]**：每条结论必须带来源，否则后处理直接拒答
> 3. **拒答兜底**：如果模型输出没引用，返回 '根据现有资料无法回答'
> 4. **Top-K 阈值**：相似度 < 0.5 的片段不喂给模型，避免被无关信息带偏
> 5. **离线评测集**：30 题人工标注，每次改动跑一遍，幻觉率 6.2% → 0.7%"

### Q19：你怎么评估 RAG 系统？

> **3 个核心指标**：
>
> 1. **Recall@K**：标准答案在 Top-K 检索结果中的比例
> 2. **Faithfulness**：答案是否完全基于 context（用 GPT-4 当评委）
> 3. **Answer Relevance**：答案是否回答了问题（也用 GPT-4）
>
> **工程实操**：
> - 30 题人工评测集（不在训练集里）
> - 每次改动（chunk_size/embedding/重排）都跑一遍
> - 守一个简单看板：4 项指标的趋势

### Q20：你怎么控制 RAG 系统的成本？

> **5 招组合**：
>
> 1. **多模型路由**：简单问 Qwen-turbo（1 元/M），复杂问 GPT-4o（30 元/M）
> 2. **缓存命中**：相似问题（query embedding 余弦 > 0.95）直接复用历史回答
> 3. **Embedding 缓存**：query embedding 算一次缓存 24h
> 4. **本地 Embedding**：用 bge-m3 本地跑，省云上 embedding 调用
> 5. **限流 + 配额**：每用户每天 100 次免费，超了走付费
