## 一、缓存 ⭐⭐

### Q1：LLM 应用怎么做缓存？三层架构

**60 秒答**：

> | 层级 | 缓存对象 | 命中条件 | TTL | 命中率 |
> |------|---------|---------|-----|--------|
> | L1 精确缓存 | query → answer | query 完全相同 | 1h | 5-10% |
> | **L2 语义缓存** ⭐ | query → answer | query embedding 余弦 > 0.95 | 24h | 20-40% |
> | L3 Embedding 缓存 | text → embedding | text 完全相同 | 7d | 50%+ |
>
> **L2 是核心**：几乎相同语义的问题直接复用答案，**省 80% LLM 调用费**。

**实现要点**：

```java
public Optional<String> hit(String query) {
    float[] emb = embeddingModel.embed(query);
    
    // 1. 精确缓存
    String exactHit = redis.get("llm:exact:" + md5(query));
    if (exactHit != null) return Optional.of(exactHit);
    
    // 2. 语义缓存：在向量库里找相似 query
    var similar = cacheStore.similaritySearch(
        SearchRequest.query(query).withTopK(1).withSimilarityThreshold(0.95)
    );
    if (!similar.isEmpty()) return Optional.of(similar.get(0).getText());
    
    return Optional.empty();
}
```

### Q2：什么场景不能用语义缓存？

> **3 类禁区**：
>
> 1. **多轮对话**：上下文不同，相同问题答案不同
>    - 解：cache key = `query` + `user_id` + `last_3_turns_hash`
> 2. **时效性问题**："今天天气" / "现在股价"
>    - 解：黑名单关键词不缓存
> 3. **个性化问题**："给我推荐一个餐厅"
>    - 解：cache key 加 `user_id`

## 二、限流与降级 ⭐⭐

### Q3：LLM 应用为什么必须限流？

**60 秒答**：

> 4 个理由：
>
> 1. **成本爆炸**：单用户狂请求 1000 次/分钟 = 一晚上烧光月度预算
> 2. **上游限流**：OpenAI/通义都有 RPM/TPM 限制，超了 429 错误
> 3. **DoS 防御**：恶意用户用免费用户角色刷你的付费 API
> 4. **公平性**：50 个用户共享配额，得有公平调度

### Q4：限流方案对比

| 维度 | 计数器 | 滑动窗口 | 令牌桶 | 漏桶 |
|------|--------|---------|--------|------|
| 实现 | 简单 | 中 | 中 | 中 |
| 临界突刺 | 有 | 无 | 无 | 无 |
| 突发流量 | 不允许 | 部分允许 | **允许** | 不允许 |
| LLM 推荐 | | | ✅ | |

**生产推荐**：**令牌桶**（用 Resilience4j RateLimiter / Sentinel / 自实现 Redis 版）。

**多维限流**：

```java
@RateLimiter(name = "llm", fallbackMethod = "rejected")
public String chat(String userId, String tenantId, String q) {
    // 三层桶：用户 / 租户 / 全局
    if (!userBucket(userId).tryAcquire()) throw new RateLimitException("user");
    if (!tenantBucket(tenantId).tryAcquire()) throw new RateLimitException("tenant");
    if (!globalBucket().tryAcquire()) throw new RateLimitException("global");
    return chatClient.prompt(q).call().content();
}
```

### Q5：模型降级方案？

> **4 级降级**（成本与质量的妥协）：
>
> ```
> 主链路：GPT-4o（优质 + 贵）
>     ↓ 失败 / 超时
> 降级 1：通义 Plus（中等 + 便宜）
>     ↓ 失败 / 超时
> 降级 2：Ollama 本地 Qwen2.5-7B（自己机器，0 成本）
>     ↓ 失败
> 降级 3：基于 ES 的纯检索 + 模板回答（无 LLM）
>     ↓
> 最终兜底：返回"系统繁忙，已转人工"
> ```
>
> **实战**：用 Resilience4j CircuitBreaker，错误率超 50% 自动跳到下一级。

## 三、成本控制 ⭐⭐

### Q6：怎么控制 LLM API 成本？

**60 秒答**：

> **6 招组合拳**：
>
> 1. **多模型路由**（最有效）：简单问 → 便宜模型，复杂问 → 贵模型
> 2. **语义缓存**（次有效）：命中率 30%+ 直接省成本
> 3. **Prompt 压缩**：去重历史、总结早期对话
> 4. **流式 + 提前终止**：用户看够了就停（省 generation 成本）
> 5. **本地 Embedding**：bge-m3 自部署，省 90% embedding 钱
> 6. **配额制**：免费 100 次/天，超了走付费

### Q7：怎么实时统计 Token 用量与成本？

**60 秒答**：

> Spring AI 用 **`Usage` 对象**：

```java
ChatResponse resp = chatClient.prompt(q).call().chatResponse();
Usage usage = resp.getMetadata().getUsage();
log.info("[METRIC] promptTokens={}, completionTokens={}, total={}",
    usage.getPromptTokens(),
    usage.getCompletionTokens(),
    usage.getTotalTokens()
);
```

**生产做法**：自定义 `UsageRecorderAdvisor` 拦截每次调用：

```java
public class UsageRecorderAdvisor implements CallAroundAdvisor {
    public AdvisedResponse aroundCall(AdvisedRequest req, CallAroundAdvisorChain chain) {
        long start = System.currentTimeMillis();
        AdvisedResponse resp = chain.nextAroundCall(req);
        Usage u = resp.response().getMetadata().getUsage();
        
        meterRegistry.counter("llm.tokens", "model", model, "type", "prompt")
            .increment(u.getPromptTokens());
        meterRegistry.counter("llm.tokens", "model", model, "type", "completion")
            .increment(u.getCompletionTokens());
        meterRegistry.timer("llm.latency", "model", model)
            .record(System.currentTimeMillis() - start, TimeUnit.MILLISECONDS);
        
        // 写 ClickHouse 离线分析
        usageStore.save(req.userParams().get("user_id"), model, u, latency);
        return resp;
    }
}
```

**4 个维度看板**：模型、用户、业务功能、时段。

## 四、监控指标 ⭐

### Q8：LLM 应用应该监控哪些指标？

**60 秒答**：

> 5 大类：
>
> ```
> ┌─ 1. 业务指标
> │   - QPS / 调用量
> │   - 端到端延迟（P50/P95/P99）
> │   - 首字延迟（TTFT）⭐ 流式独有
> │   - 错误率
> │
> ├─ 2. 模型质量
> │   - Token 用量（in/out 分别）
> │   - 缓存命中率
> │   - Tool 调用成功率
> │   - 拒答率（防幻觉）
> │
> ├─ 3. 成本
> │   - 每次调用成本（元）
> │   - 每用户日均成本
> │   - 每模型成本占比
> │
> ├─ 4. RAG 专属
> │   - 检索召回率（离线评测）
> │   - 平均 Top-K score
> │   - 文档片段命中分布
> │
> └─ 5. Agent 专属
>     - 平均迭代次数
>     - Tool 调用分布
>     - 任务完成率
> ```

### Q9：流式延迟怎么监控？

> 流式有 3 个关键时间点：
>
> ```
> 用户发请求
>     ↓ T1：网关接收
>     ↓ T2：模型首字（TTFT）⭐
>     ↓ T3：每个 chunk 间隔（throughput）
>     ↓ T4：完整结束
> ```
>
> **关键指标**：**TTFT P95 < 800ms**（用户感知"快"的红线）。

## 五、安全 ⭐⭐

### Q10：Prompt Injection 怎么防？

**60 秒答**：

> 5 道防线（详见 [`04-Agent与工程化/笔记/PromptInjection防御.md`](../../04-Agent与工程化/笔记/PromptInjection防御.md)）：
>
> | 防线 | 位置 | 拦截 |
> |------|------|------|
> | 1 输入清洗 | 入口 | 直接攻击关键词 |
> | 2 角色锁定 | system prompt | "忘掉之前指令"类 |
> | 3 工具白名单 | Tool 调用 | 越权调用 |
> | 4 输出审查 | 返回前 | 泄露 system prompt |
> | 5 限流 + 监控 | 旁路 | 暴力枚举 |
>
> **效果**：20 个攻击样本测试，95%+ 拦截率。

### Q11：怎么防止数据泄露？

> **3 个层面**：
>
> 1. **数据层**：敏感字段 Embedding 前脱敏（手机号 → `139****1234`）
> 2. **检索层**：基于 user_id / tenant_id 的元数据过滤，**绝不能跨租户**
> 3. **生成层**：output filter 扫描泄露的 PII（用正则 + NER）

```java
// VectorStore 多租户过滤
SearchRequest.query(q)
    .withFilterExpression("tenant_id == '" + tenantId + "' AND user_id == '" + userId + "'")
    .withTopK(5);
```

⚠️ **安全红线**：**filter 必须服务端添加，不能信任前端传的 tenant_id**。

## 六、A/B 测试 ⭐

### Q12：LLM 应用怎么做 A/B 测试？

**60 秒答**：

> **3 个层面可以 A/B**：
>
> 1. **模型层**：50% 用户走 GPT-4o，50% 走 Claude
> 2. **Prompt 层**：50% 用 Prompt v1，50% 用 v2
> 3. **检索层**：50% 用 chunk_size=512，50% 用 1024
>
> **流量切分**：基于 `user_id % 100` 哈希，**保证同用户始终在同组**。

### Q13：怎么衡量哪个版本"更好"？

> **离线 + 在线双指标**：
>
> ```
> 离线（开发阶段）：
>   - 30 题评测集，GPT-4 当裁判，评分 1-5
>   - 自动化 CI 跑：分数 < 4.0 的版本拦截上线
>
> 在线（生产观察）：
>   - 用户主动反馈（👍/👎）
>   - 隐式信号：是否复制答案 / 是否再追问
>   - 业务转化：客服转人工率
>   - 成本：单次调用成本
> ```

## 七、并发与性能 ⭐

### Q14：LLM 应用单机能扛多少 QPS？

> **取决于瓶颈**：
>
> ```
> 应用层（Spring Boot）：3000-5000 QPS（轻量逻辑）
>     ↓
> Embedding 调用：100-500 QPS（IO 等模型）
>     ↓
> LLM 调用：10-50 QPS（IO 等远程，主要瓶颈）
> ```
>
> **结论**：**LLM 应用 QPS 上限 = 模型上游 RPM ÷ 60**。
>
> 提升方案：
> - **批处理**：积攒 16 个请求一次发（Embedding 适用）
> - **多模型分散**：通义 + GPT-4o + Ollama 三路并行，QPS 翻 3 倍
> - **缓存命中**：30% 命中率 = 实际 QPS 翻 1.4 倍

### Q15：响应慢的 7 个排查点？

```
1. 网络延迟（curl --trace 看 DNS / TCP / TLS 各阶段）
2. Prompt 太长（统计 token 数，超过 4k 慢）
3. 模型本身慢（小模型快但差，大模型准但慢）
4. Embedding 调用串行（应该批量 + 缓存）
5. 检索慢（向量库索引没建好 / shard 不均）
6. Tool 调用同步（多个 Tool 应该并行）
7. 网关 buffering（Nginx proxy_buffering 必须 off）
```

## 八、可靠性 ⭐

### Q16：LLM 调用怎么做重试？

> **关键原则**：**只重试可恢复错误**。
>
> | 错误码 | 应对 |
> |--------|------|
> | 429 限流 | **重试**（指数退避 1s/2s/4s）|
> | 500/502/503 服务异常 | **重试**（指数退避）|
> | 504 超时 | **重试一次** |
> | 401 鉴权失败 | **不重试**（直接报警）|
> | 400 参数错 | **不重试**（修代码）|
> | 上下文超限 | **不重试**（截断后再发）|
> | 内容审核拒绝 | **不重试**（返回友好提示）|

**Spring AI 写法**（用 Resilience4j）：

```java
@Retry(name = "llm", fallbackMethod = "fallback")
@CircuitBreaker(name = "llm", fallbackMethod = "fallback")
public String chat(String q) {
    return chatClient.prompt(q).call().content();
}

public String fallback(String q, Exception e) {
    log.warn("LLM failed, fallback to local", e);
    return ollamaClient.prompt(q).call().content();  // 降级本地
}
```

### Q17：Tool 调用失败怎么处理？

> 3 层兜底：
>
> 1. **Tool 内部 try-catch**：返回 `{"error": "服务暂不可用，请稍后重试"}` 给模型
> 2. **重试策略**：可重试错误重 1-2 次（不要重太多 → 雪崩）
> 3. **Agent 容错**：把错误信息塞回 prompt，让模型决定换路（"工具失败，要不要换一个"）
