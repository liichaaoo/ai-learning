# RAG 调优技巧（速查手册）

> 服务于 Week 4-6。**3 个月后回看，调优清单一目了然。**

---

## 1. 一句话总纲

> **RAG 效果 ≈ 检索质量 × Prompt 设计 × 模型能力**——其中**检索占 70% 权重**。

---

## 2. 调优旋钮速查表 ⭐

| 旋钮 | 范围 | 调高的影响 | 工程默认 |
|------|------|-----------|----------|
| chunk_size | 200~1000 | 上下文全但召回糊 | **500** |
| chunk_overlap | 0~200 | 防断头但冗余 | **50~100** |
| top_k | 3~10 | 全但噪声多 | **5** |
| similarity_threshold | 0~1 | 严但易拒答 | **0.5** |
| temperature | 0~1 | 多样但易飘 | **0**（RAG）|
| max_tokens | 256~2000 | 长但贵 | **512** |
| reranker | on/off | 准但慢 | **on**（生产）|

## 3. 我的真实调优实验数据（基于 30 题评测集）

| 配置 | Top-3 召回率 | MRR | P95 延迟 |
|------|-------------|-----|---------|
| 纯向量 topK=3 | 65% | 0.55 | 1.1s |
| **纯向量 topK=5** | **80%** | **0.70** | **1.2s** |
| **+ 混合检索（向量+BM25+RRF）** | **88%** | **0.82** | **1.3s** |
| **+ Reranker（gte-rerank）** | **90%** | **0.85** | **1.5s** |
| chunk=300 | 70% | 0.58 | 1.2s |
| chunk=800 | 75% | 0.62 | 1.3s |

> 🎯 **简历叙述**：「Top-3 召回率从 65% 提升至 90%，MRR 从 0.55 → 0.85」

## 4. 阶梯式调优清单（按效果性价比排序）

```
1. 先把 chunk_size / overlap 调对（最大杠杆）
   500 / 100 是绝大多数场景的甜点
   特别长的文档可以试 800 / 150

2. 加 similarity_threshold = 0.5
   把"凑数"的 chunk 砍掉，质量立竿见影

3. 上混合检索（向量 + BM25 + RRF）
   特别在精确词检索场景（订单号、人名）效果暴涨

4. 加 Reranker（gte-rerank / bge-reranker-v2-m3）
   Top-K 精度再涨一档，但延迟 +200-500ms

5. system prompt 防幻觉：找不到时拒答
   降低"自信胡说"概率

6. metadata 过滤
   时间衰减、租户隔离、文档类型筛选

7. HyDE 查询改写
   长尾问题召回率 +X%

8. 答案缓存
   FAQ 类高频问题直接命中缓存
```

## 5. 出问题怎么排查

| 现象 | 第一反应 | 排查步骤 |
|------|---------|---------|
| 答案完全跑题 | 检索没召回 | 看 vector_store.search 实际命中了啥 |
| 答案部分对部分错 | top_k 太大 | 减到 3 / 看 chunk 分割是否切到关键句子 |
| 答案"自信地编造" | system prompt 弱 | 加"找不到就拒答"的硬约束 |
| 拒答太多 | 阈值太高 | 0.7 → 0.5 |
| 延迟突然飙升 | 看是哪一段 | embed / 召回 / 重排 / LLM 各打日志 |
| 内存暴涨 | chunks 过多 | 检查 `chunkRepo.countByDocumentId()` 是否合理 |

## 6. Prompt 工程模板（防幻觉）

```
你是公司知识库助手，必须严格遵守以下规则：

【规则】
1. 只能基于"参考文档"回答用户问题。
2. 如果参考文档没有相关信息，直接回答"我没有在公司知识库找到相关信息"。
3. 不要编造文档中不存在的事实、人名、数字、日期。
4. 答案末尾必须列出引用 [n] 标号。
5. 用简洁、专业的中文回答。

【参考文档】
{documents}

【用户问题】
{question}

【回答】
```

## 7. 性能优化清单（按性价比）

```
1. Embedding 缓存：完全相同的文本不要重复 embed
2. 批量 embed：32~64 一批
3. KV Cache：流式响应已默认开启
4. 答案缓存：高频 FAQ 走 Redis（key=问题哈希）
5. 异步入库：上传不阻塞，跑队列
6. Reranker 异步：召回先返回，重排在后台
7. 模型路由：FAQ 用 qwen-plus，复杂用 qwen-max
8. CDN / 静态资源：前端永久缓存
```

## 8. 评测体系（简历核心数字来源）

```
评测集（30+ 题）
  ├─ 简单事实题（Recall 主要在这）
  ├─ 多文档融合题（看是否跨文档理解）
  ├─ 拒答题（库里没的，看是否拒答）
  └─ 模糊查询题（"那个东西"，看检索鲁棒性）

指标：
  Recall@K        Top-K 命中率
  MRR             命中位置倒数均值
  Keyword Hit%    答案是否包含期望关键词
  RejectAcc       拒答正确率
  Latency P95     性能
```
