## 1. 简历模板（直接套用，改自己的数）

```markdown
**企业 RAG 知识库系统 [Java / Spring AI / Milvus]**（个人项目，开源 GitHub）

- 主导设计基于 Spring AI + Milvus 的企业 RAG 知识库系统，
  覆盖 200+ 内部文档（PDF/Word/Markdown），支持多租户隔离。
- 实现 **混合检索（向量召回 + BM25 + Rerank）**，Top-3 召回率从
  65% 提升至 90%（基于 30 题人工评测集），MRR 从 0.55 → 0.85。
- 设计 **基于场景的多模型路由策略**（通义 / GPT-4o / Ollama），
  按成本 / 质量 / 隐私自动选择，月度推理成本节省 ~40%。
- 基于 Spring AI Advisor 链实现 **Token 用量统计、成本告警、
  审计日志、敏感词过滤** 等横切能力。
- 异步入库流水线（RabbitMQ + 进度查询），单实例支撑
  入库吞吐 1500 chunks/min。
- **k6 压测：50 并发下单实例 P95 延迟 < 1.6s，错误率 < 0.1%**；
  Docker Compose 一键部署。
- GitHub: github.com/yourname/enterprise-rag-kb
```

## 2. 数字怎么"采集"——分维度查表

### 2.1 性能数字

| 指标 | 工具 | 怎么得 |
|------|------|--------|
| QPS | k6 / wrk / ab | 见 Week5-6 阶段⑨ |
| P50 / P95 延迟 | 同上 | 同上 |
| 首字延迟（流式）| 自己埋点 | 流式输出第一个 chunk 时间 |
| 入库吞吐 | 写日志统计 | total_chunks / total_seconds |
| 错误率 | k6 输出 | http_req_failed |

### 2.2 质量数字

| 指标 | 怎么得 |
|------|-------|
| 召回率 | 30+ 题评测集，跑自动化脚本，见阶段⑦ |
| MRR | 同上 |
| 关键词命中率 | 同上 |
| 拒答正确率 | 故意问知识库没有的题，看是否拒答 |

### 2.3 成本数字

| 指标 | 怎么得 |
|------|-------|
| Token 总消耗 | usage_record 表 SUM |
| 成本节省 % | (路由前模拟成本 - 路由后实际成本) / 路由前成本 |
| 单次平均成本 | 总成本 / 调用次数 |

### 2.4 工程数字

| 指标 | 怎么得 |
|------|-------|
| 文档量 | document 表 count |
| chunk 数 | chunk 表 count |
| 测试覆盖率 | jacoco（如果跑了）|
| 部署时间 | docker compose up 计时 |

## 3. 简历 Bullet 写法的 4 种模式

### 模式 A：技术 + 数字
> "实现混合检索（向量+BM25+Rerank），**Top-3 召回率 65% → 90%**"

### 模式 B：方法 + 收益
> "通过多模型路由策略，**月度推理成本节省 ~40%**"

### 模式 C：规模 + 性能
> "覆盖 200+ 文档，**单实例 P95 延迟 < 1.6s，支撑 50+ QPS**"

### 模式 D：能力 + 复用
> "基于 Spring AI Advisor 链实现 **Token 计费、成本告警、审计日志** 等横切能力，可在多项目复用"

## 4. 数字"可信"的几个原则

```
✅ 必须：有具体数字（"提升 30%"而非"显著提升"）
✅ 必须：能讲清楚怎么算的（面试官真会追问）
✅ 必须：跟项目规模匹配（个人项目说 10 万 QPS 就别玩了）
⚠️ 建议：括号里附"基于 X 题评测集" / "基于 X 万次调用"
⚠️ 建议：百分比 + 绝对值都给（"省 40%（约 ¥2000/月）"）
❌ 避免：吹"业内领先" / "性能最优" 这种没法证明的话
❌ 避免：随口编一个 99.9% / 1ms 这种太完美的数字
```

## 5. 面试官最爱问的"数字背后"

| 提问 | 怎么答 |
|------|-------|
| **"Top-3 召回率怎么测的？"** | 写了 30 题评测集 + 自动化脚本，每题标注期望来源，命中即算 |
| **"为什么是 30 题不是 100 题？"** | 个人项目时间有限，30 题已经能覆盖 7 大场景；面试时坦诚说 |
| **"成本节省 40% 是怎么算的？"** | 用基线（全部走 qwen-max）算月理论成本，对比实际路由后成本 |
| **"P95 1.6s 在哪里测的？"** | k6 50 并发 + 标准评测 prompt，本机 M1 / Linux 服务器都行 |
| **"为什么不上 GPT-4？"** | 个人 API Key 成本敏感 + 国内访问 + 通义中文表现已够；架构上预留接口 |

## 6. 数字之外的"故事化"亮点

数字之外还可以讲：

- **"踩坑故事"**：chunk_size 选错导致召回崩 → 评测对比 → 找到甜点（500/100）
- **"权衡决策"**：Reranker 加不加，对比 +200ms 延迟 vs +5pp 召回率，选择加
- **"工程取舍"**：评测集 30 题不是 1000 题，为什么够用
- **"架构演进"**：从 Week 4 mini 到 Week 5-6 完整版，加了哪些能力、为什么

> 🎯 **故事让数字有了温度**——同样的数字，会讲故事的人能多拿 30% 印象分。
