## 1. 我（们）的选择：Milvus

> **理由 30 秒能讲完（面试常问）**：
>
> 1. **专业向量库**：HNSW / IVF / GPU 索引一应俱全，性能上限高
> 2. **Spring AI 一等公民**：starter 开箱即用
> 3. **生态成熟**：Java SDK / Attu GUI / Zilliz 云托管
> 4. **数据量友好**：百万到上亿级都能扛
> 5. **国内生态好**：文档齐全，社区活跃，腾讯/字节都在用

## 3. 选型决策树

```
你已经有 PG 了？
  ├─ 是 → 数据 < 100万？
  │       ├─ 是 → ✅ PgVector（运维零负担）
  │       └─ 否 → 考虑 Milvus
  │
  └─ 否 → 你已经有 ES 了？
         ├─ 是 → 想做全文 + 向量？

         │       └─ 否 → Milvus
         │
         └─ 否 → 数据量？
                ├─ < 10万 / 学习  → ✅ Chroma 或 Milvus Lite
```

## 4. Milvus 工程默认参数

```yaml
# Spring AI 配置
spring.ai.vectorstore.milvus:
  collectionName: your_collection
  embeddingDimension: 1024            # ⚠️ 跟模型对齐
  indexType: HNSW                      # 工业默认
  metricType: COSINE                   # Embedding 默认归一化时与 IP 等价
  initialize-schema: true              # 启动自动建库
```

```text
# HNSW 三参数
M = 16                  邻居连接数（增大 → 准但慢，工程默认 16）
efConstruction = 200    建索引探索数（256 也行，建一次）
ef（运行时）= 64        查询探索数（默认 64，64-128 折中）
```

## 5. 关键工程踩坑

| 坑 | 表现 | 解法 |
|----|------|------|
| **维度不匹配** | 写入直接 500 | Embedding 模型维度 = Collection dimension |
| **没建索引就查** | 查询超慢 | `createIndex` 后 `loadCollection` |
| **没 load 就查** | "collection not loaded" | `loadCollection` |
| **Partition 滥用** | 性能反而下降 | 不超百万数据不用 Partition |
| **删除 + 立即查** | 还能查到旧数据 | Milvus 是异步删除，需要等 compaction |
| **大批量写入慢** | 写一个一个调 SDK | 用 batch insert，1000~5000 条一批 |

---

## 6. 切换 VectorStore 是 0 成本吗？

**Spring AI 抽象做得好**，但有 3 件事会"硬变"：

1. **维度限制** —— 不同 store 的 max dimension 不同
2. **filter 表达式语法** —— Milvus / PgVector / Chroma 各自不一样
3. **元数据类型** —— 比如 Milvus 早期版本对 dynamic fields 支持有限

> 🎯 **结论**：切换不是"改一行依赖"，**但接口层（VectorStore）不用改**——业务代码可以无痛迁移。
